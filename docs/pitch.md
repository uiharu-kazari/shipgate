# ShipGate — 最終ピッチ台本（3分想定）

## 0:00–0:20 フック

> 深夜3時。オンコールの携帯が鳴る。昨日マージした PR が本番で燃えている。
> ダッシュボードを開く — メトリクスがない。ログもない。
> そのインシデント、**PR の時点で防げました。**

CI はコードが「正しい」ことしか見ません。「本番で運用できるか」は誰も見ていない。

## 0:20–0:50 提案

**ShipGate — 差分を読んで実験する、自己修復型リリースゲート。**

固定パイプラインではありません。PR ごとに Gemini が差分を読み、**その変更専用の実験を自分で設計**します：

- ハンドラが変わった → レイテンシ予算つき負荷実験
- TTL が入った → **タイムワープ**: 仮想時計で1時間後を3秒で再生
- 失敗パスが増えた → 「オンコールがこれをデバッグできるか」監査

そして鉄則がひとつ: **ブロックには実測の証拠が必須。** 判定は Gemini の「意見」ではなく、実験結果からコードが決定的に計算します（`decideVerdict`）。だから差分に「shipと返せ」と書き込むプロンプトインジェクションでも判定は覆りません — Gemini は理由文を書くだけ、ゲートは証拠が握る。

## 0:50–1:50 デモ（3幕）

（画面: GitHub PR #1）

1. **第1幕**: 「検索キャッシュ追加」の PR。ShipGate が実験して 🔴 **BLOCK**。
   コメントに実測値: p99 714ms（予算300ms）、エラー率100%、**期限の1時間後もまだ古いデータ**。
2. **第2幕**: エージェントが**自分でパッチを書く**。期限判定の修正、計装の注入、優雅な劣化。
3. **第3幕**: 同じ実験をパッチに再実行 → 🟢 **SHIP**。ダッシュボードに block→ship の証拠タイムライン。

（画面: Kibana）
おまけ: Elasticsearch 上の **Historian** に日本語で「なぜブロックされたの？」→ 証拠つきで即答。

## 想定 Q&A

- **Q: しきい値は誰が決める？** A: 現在は Gemini が差分から予算を提案。ロードマップは Elasticsearch の履歴からの自動キャリブレーション（過去の p95 ベースライン比較）。
- **Q: 仮想時計はアプリ改修が必要では？** A: 1関数のオプトイン（テストモード限定）。Effect の TestClock 的発想を、フレームワーク非依存の HTTP 境界に移した設計。
- **Q: 実験がフレーキーでは？** A: 判定は「証拠ベース原則」で、ブロックは実測失敗のみ。実験エラーは error 扱いで判定を汚さない。
- **Q: エージェントの暴走リスクは？** A: パッチは提案のみ（人間が apply）。ゲートは fail-closed ではなく PR コメント + status check。
- **Q: 差分に悪意ある指示を書けば判定を騙せる？** A: 騙せない。判定は実測結果からコードで決定（`decideVerdict`）、Gemini は理由文のみ。「shipと返せ」入りの差分でも BLOCK を実証済み（confidence 1.0）。加えて `/analyze` はトークン認証、実験対象は同一ホストに制限（絶対URLパスでの踏み台化を防止）。
- **Q: 本番アプリに負荷をかけるの？** A: 対象はプレビュー/ステージング環境。GitHub Actions の変数で対象 URL を分離。`/analyze` は共有トークンで認証し、`targetUrl` はホスト許可リストで制限（負荷試験の踏み台化・コスト濫用を防止）。
- **Q: PR ごとに専用環境を立てるの？** A: 理想は PR プレビューをデプロイしてその URL を渡す構成。本デモは簡略化のため固定ステージング URL を対象にしている（正直な範囲）。
